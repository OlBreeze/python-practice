# Обзор лекции по Python Pro

## Проект студента: E-commerce платформа

### Архитектура проекта
- **Микросервисная архитектура** с использованием Docker контейнеров
- **Сервисы:**
  - Сервис D (Аутентификация)
  - Сервис E (Уведомления/Telegram бот)
  - Сервис A (Каталог/Инвентарь)
  - Order сервис

### Технологический стек
- **Backend:** FastAPI
- **БД:** PostgreSQL (асинхронная работа через ASGP)
- **Кеширование:** Redis
- **Авторизация:** JWT токены с дополнительной криптографией
- **Контейнеризация:** Docker, Docker Compose
- **Будущие планы:** Kubernetes, Nginx

### Система аутентификации
**Особенность:** Авторизация через Telegram-бот вместо SMS (экономия средств)

**Процесс:**
1. Пользователь делится номером телефона в Telegram
2. Система связывает номер с Telegram chat_id
3. Генерируется 6-значный OTP код
4. Код отправляется в Telegram (TTL 5 минут)
5. После верификации выдаются access и refresh токены

**Проблема на момент лекции:** Ошибка "integer must be string" при работе Redis consumer

### Что нужно улучшить в проекте:

1. **Безопасность:**
   - Добавить логин/пароль (номер телефона как единственный идентификатор небезопасен)
   - Реализовать Zero Trust архитектуру
   - Настроить Nginx для обработки запросов и защиты от DDoS

2. **Диверсификация системы уведомлений:**
   - SMS (для критичных операций)
   - Email
   - Telegram
   - Viber, WhatsApp (опционально)
   - **Важно:** Минимум 2-3 способа связи для восстановления доступа

3. **Документация:**
   - README файлы
   - Понятные комментарии
   - ADR (Architecture Decision Records) - документация почему выбрано конкретное решение

4. **Оптимизация:**
   - Использовать профайлеры и тестеры нагрузки
   - Симуляция 1000+ одновременных подключений
   - Конкретизировать Exception блоки (не использовать общий except)
   - Использовать транзакции, изучить ACID

5. **Базы данных:**
   - Комбинация SQL (PostgreSQL) + NoSQL
   - Шардинг для распределения нагрузки
   - Elasticsearch для каталога и фильтрации
   - Redis для кеширования (поисковые запросы, данные каталога)
   - Бэкапы!

6. **CI/CD:**
   - Линтеры (Black, Flint)
   - Автоматическое тестирование
   - Пайтесты перед коммитами
   - Настроить проверки в GitHub перед пушем

7. **Health checks:** Обязательны для Kubernetes (критично важно!)

## Советы по поиску работы

### Подготовка к собеседованиям:

**Изучить обязательно:**
- ACID принципы (часто спрашивают!)
- Паттерны проектирования
- Транзакции
- CI/CD процессы

**На собеседовании:**
- В среднем 5-10 собеседований до оффера
- Изучить компанию заранее (чем занимается, какие домены)
- Уточнить технологический стек
- **Задавать свои вопросы:**
  - Размер команды?
  - Есть ли онбординг/менторинг?
  - Есть ли roadmap развития?
  - Как организована коммуникация?
  - К кому обращаться с вопросами?
  - Интернациональная команда или нет?

**Про тестовые задания:**
- Если делаете 10+ тестовых - начинайте спрашивать про оплату
- Не стесняйтесь говорить "не знаю" - лучше честно признаться

**Красные флаги:**
- Нет четкого описания задач
- Один ходит на созвоны, другой пишет код
- Отсутствие структуры и документации
- HR не может ответить на технические вопросы
- Требуют натрекать 8 часов в трекере (убивает продуктивность)

## Git культура (из опыта работы с китайской командой)

### Строгие правила коммитов:
- Формат: `[feature/bug] #номер_задачи описание (минимум 80 символов)`
- Пример: `feature #23 add validation for phone numbers in registration form`
- Четкая структура Pull Request
- Обязательные ревьюеры
- Линтеры встроены в репозиторий
- Все изменения документируются

**Плюсы такого подхода:**
- Легко откатиться к нужному коммиту
- Понятно кто, что и когда менял
- Меньше хаоса в команде
- В долгосрочной перспективе экономит время

## Философия обучения

### Главный принцип:
**Учиться не конкретному инструменту, а концепциям!**

- MVC везде одинаковая концепция
- Циклы работают одинаково (отличается только синтаксис)
- HTTP запросы формируются по тем же принципам
- Паттерны проектирования универсальны

**Аналогия:** Как строительство дома - неважно из кирпича или модулей, концепция одна: фундамент → стены → крыша.

Когда понимаешь концепции, переход на новый язык/фреймворк занимает недели, а не месяцы.

## Практические советы

1. **Не останавливаться в развитии:** Читать книги, изучать смежные области (frontend, DevOps)
2. **Практика постоянно:** Делать pet-проекты по выходным, даже если есть работа
3. **Разнообразие опыта:** Вероятно сменишь язык программирования несколько раз за карьеру
4. **Fullstack перспектива:** Знание фронтенда (React, Vue, TypeScript) - большой плюс
5. **Bash для DevOps** - не так сложно, как кажется
